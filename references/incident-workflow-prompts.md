# 线上故障 / 紧急问题流程模板

本文件用于生产环境故障、告警、用户受影响、资损风险、数据错误、任务堆积、接口大量失败、第三方异常、权限泄露、发布事故和紧急回滚。核心是先止血、判断影响范围、保护数据和留存证据，再定位根因、修复、补偿和复盘。

## 使用规则

- 第 1 到第 4 段：以止血、证据收集、影响评估和方案确认优先，不直接改代码。
- 第 5 段：只执行一个明确的 `M` 止血、修复或补偿点。
- 第 6 段：基于实际 `diff`、线上证据、影响面和 `P/M/T` 做 review，并输出验证、补偿和复盘清单。
- 任何可能删除数据、重置代码、执行补偿脚本、影响外部系统或扩大线上影响的操作，必须先获得用户明确确认。
- 没有监控、日志、数据、告警、请求样例或代码证据的地方写“未确认”，不要猜成事实。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把六段一次性全部执行。

## 第 1 段：故障分级与止血判断

适用时机：用户刚提出线上故障、告警、生产问题、客户受影响、数据错乱或紧急异常，还没有确认影响范围和止血策略。

```text
请使用 $goal-path-analysis。
先不要改代码，不要执行破坏性命令，不要修改线上数据。

我遇到一个线上故障 / 紧急问题：
【粘贴告警、现象、时间、影响用户、接口、日志、截图、监控或当前处理进展】

请先帮我整理：

1. 故障现象
- 当前线上表现是什么？
- 是页面不可用、接口失败、数据错误、任务堆积、消息积压、性能劣化、权限问题、资损风险还是第三方异常？
- 首次发现时间、持续时间、影响环境是什么？

2. 影响分级
- 影响哪些用户、租户、订单、金额、任务、接口或外部系统？
- 是否有资损、数据损坏、权限泄露或不可逆副作用风险？
- 是否仍在扩大？

3. 止血选项
- 是否需要回滚、关开关、降级、限流、暂停任务、暂停消费、隔离入口或切换配置？
- 每个止血选项的收益、风险和回滚方式是什么？

4. 证据保全
- 需要保留哪些日志、traceId、请求样例、数据库快照、消息 offset、配置版本、发布版本或监控截图？
- 哪些证据已经有，哪些还缺？

5. 需要立刻确认的问题
- 哪些信息不确认会影响止血方案？
- 哪些操作需要用户明确授权？

只分析，不要改代码。
没有证据的地方请写“未确认”，不要猜成事实。
如果存在持续扩大、资损、数据损坏或权限泄露风险，请优先给出最小止血建议，并明确哪些操作需要确认。
```

合格输出必须让用户看清：故障级别、影响范围、是否仍在扩大、可选止血动作、证据保全和需要授权的操作。

## 第 2 段：影响范围与入口路径对账

适用时机：已初步判断故障现象，需要把影响面、入口路径和扩散边界查清楚。

```text
请继续使用 $goal-path-analysis。
请基于当前代码、日志、监控、配置和数据证据对账影响范围与入口路径，仍然不要改代码。

请先列出：
1. 你搜索过的关键词
2. 你检查过的文件、配置、日志、监控或数据
3. 你命中的类名、方法名、SQL、配置、接口、任务、topic、traceId 或日志关键字

然后按这个格式输出：

1. 受影响入口
- 页面入口
- 接口入口
- 定时任务入口
- MQ / Event / Webhook 入口
- 批量任务 / 导入导出入口
- 管理后台或脚本入口

2. 扩散路径
- 哪些路径会继续写数据、发消息、调外部系统或影响用户？
- 哪些路径已经停止？
- 哪些路径还未确认？

3. 数据影响
- 受影响表、字段、状态、枚举、金额、库存、权限或任务状态是什么？
- 是否需要冻结、备份、对账或补偿？

4. 外部影响
- 是否影响第三方平台、支付、ERP、仓库、通知、Webhook 或下游系统？
- 是否需要通知对方或暂停调用？

5. 排除项
- 哪些入口已检查未受影响？
- 证据是什么？

每个结论都必须带证据位置：文件路径 / 方法名 / SQL / 配置 / 日志 / 监控。
没有证据的地方请写“未确认”，不要猜成事实。
```

合格输出必须能回答“影响谁、影响多少、是否还在扩大、哪些入口已排除、哪些未确认”。

## 第 3 段：根因定位与处置方案

适用时机：止血选项和影响面明确后，需要定位根因并选择修复、回滚或补偿方案。

```text
请继续使用 $goal-path-analysis。
请基于线上证据、影响路径和代码调用链做根因定位，并给出处置方案。先不要改代码，不要执行线上操作。

请输出：

1. 当前状态判断
- 是否已经止血？
- 是否仍在扩大？
- 是否需要先执行止血再继续定位？

2. 根因判断
- 最可能根因是什么？
- 证据是什么？
- 还有哪些候选根因？
- 哪些已排除，为什么？
- 哪些仍未确认？

3. 处置方案候选
- 方案 A：止血优先方案
- 方案 B：代码修复方案
- 方案 C：数据补偿或修复方案
- 方案 D：回滚或降级方案

4. 推荐方案
- 推荐先做哪一步？
- 为什么？
- 需要哪些授权？
- 风险和回滚方式是什么？

5. 数据保护
- 是否需要备份、快照、只读核对、补偿脚本预演或人工复核？
- 哪些数据不能直接改？

只分析，不要改代码。
如果没有足够证据确认根因，请不要进入修复阶段，先说明还需要什么日志、数据、监控或复现样例。
```

合格输出必须把止血、根因、修复、回滚和数据补偿拆开，不把高风险操作混成一次修改。

## 第 4 段：生成 P/M/T 处置账本

适用时机：根因和处置方案已经初步确认，需要拆成可授权、可回滚、可验证的处置项。

```text
请继续使用 $goal-path-analysis。
请把线上故障处置方案整理成 P/M/T 账本，先不要改代码，不要执行线上操作。

P = 故障路径：
- 每条路径从哪里进入
- 调用链或扩散链是什么
- 当前错误结果是什么
- 是否仍在扩大
- 影响范围是什么

M = 处置点：
- 每个 M 是止血、修复、回滚、补偿、对账还是复盘动作
- 绑定哪些 P
- 允许修改哪些文件、配置或数据
- 不允许做什么
- 是否需要用户明确授权
- 回滚方式是什么
- 风险是什么

T = 验证项：
- 如何证明止血成功
- 如何证明修复成功
- 如何证明数据补偿正确
- 如何证明没有继续扩大
- 验证位置是页面、接口、数据库、日志、监控还是第三方平台

最后给出推荐处置顺序。
没有我的确认，不要开始改代码或执行线上操作。

最后请自检：
1. 哪些 P 还没有线上证据？
2. 哪些 M 涉及高风险操作或外部系统？
3. 哪些 T 不能证明止血、修复或补偿成功？
4. 哪些数据需要先备份或人工复核？
```

合格输出必须明确每个处置点的授权要求、回滚方式和验证信号。

## 第 5 段：单点处置

适用时机：用户确认 `P/M/T` 账本后，只允许执行一个明确的 `M`。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只处理 M【替换成真实编号】，不要处理其他 M。
例如：现在只处理 M1，不要处理 M2、M3。

动手前请先复述：
1. 本次只做什么：止血 / 修复 / 回滚 / 补偿 / 对账
2. 对应的故障路径和根因是什么
3. 是否涉及线上数据、配置、外部系统或破坏性操作
4. 是否已获得明确授权
5. 覆盖哪些 T 验证项
6. 允许修改哪些文件、配置或数据
7. 不允许修改或执行什么
8. 回滚方式是什么

然后开始处理。

完成后请说明：
1. 修改或执行了什么
2. 为什么这样处理
3. 如何对应故障根因或止血目标
4. 覆盖哪些 P/M/T
5. 已执行了哪些验证，结果是什么
6. 哪些路径、数据、用户或外部系统还没处理
7. 是否还存在扩大风险
```

合格输出必须清楚说明当前只完成哪个处置点，不得把未授权、未执行或未验证项说成已完成。

## 第 6 段：diff review + 验证补偿复盘清单

适用时机：某个 `M` 处置完成后，需要审查真实改动、确认止血或修复效果，并整理补偿和复盘事项。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff / 操作记录、线上证据、影响面和 P/M/T 账本严格 review 这次处置，只列有证据的真实风险。

如果缺少 diff、操作记录、线上证据、影响面、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名 / SQL / 配置 / 日志 / 监控 / 操作记录
影响范围：影响哪个 P/M/T、用户、数据或外部系统
风险等级：高 / 中 / 低
建议处理：

检查这些方面：
1. 是否已经止血，影响是否停止扩大
2. 是否真正修复根因，还是只是缓解现象
3. 是否遗漏入口、旁路、任务、消息或外部回调
4. 是否有数据损坏、重复补偿、少补偿、多补偿或状态不一致
5. 是否有权限、数据归属、事务、幂等、并发或重复执行问题
6. 是否有 ID 精度、金额精度或时间格式问题
7. 是否影响公共方法、配置、任务、消息或其他业务
8. 是否有监控、告警、日志或审计缺失
9. 是否具备回滚方案和二次验证路径
10. 是否需要用户通知、第三方通知、数据补偿或复盘行动项

然后输出验证 / 补偿 / 复盘清单：

清单项：
类型：止血验证 / 修复验证 / 数据补偿 / 影响面确认 / 通知 / 复盘
覆盖的 P/M/T：
前置条件：
操作步骤：
预期结果：
验证位置：页面 / 接口 / 数据库 / 日志 / 监控 / 第三方平台
是否必须执行：是 / 否
当前状态：已完成 / 未完成 / 需人工确认

要求：
1. 必须包含止血验证
2. 必须包含根因修复验证
3. 必须包含受影响数据和用户范围确认
4. 必须包含补偿或无需补偿的证据
5. 必须包含监控、告警或日志复查
6. 没有证据的清单项标记为“需人工确认”
7. 不要把“建议处理”写成“已完成”
```

合格输出必须能支持用户判断故障是否停止、根因是否修复、数据是否需要补偿、后续是否需要复盘。

## 口诀

```text
先止血
保证据
查影响
定根因
列 P/M/T
单点处置
补偿复盘
```
