# 实施细则

只有当用户要求直接改代码、补链路、修复 bug 或落实现时才读取本文件。

目标不是“一次改很多”，而是沿着已经收敛的路径账本做最小可靠修改，减少 review 后的返工。

## 1. 进入实施前的硬前提

编码前必须先有：

- 路径总表 `P1...PN`
- 修改点列表 `M1...MN`
- 验证场景 `T1...TN`

如果这些编号还不存在，先补齐再编码。禁止边写代码边临时猜“应该就这一条路径”。

## 2. 修复点设计

每个修复点在动手前都要明确：

- 绑定的路径 ID
- 本次只解决的缺口
- 计划修改的位置
- 最大改动半径
- 什么证据算修好
- 需要顺手复查的相邻路径或公共位置

其中“最大改动半径”至少写清：

- 本次允许修改的文件、类、模块或目录范围
- 是否允许触碰公共方法、公共 SQL、公共事件或共享配置
- 是否允许影响数据库结构、消息协议、外部接口或定时任务

如果某个修复点需要突破原先半径，先停下更新 `M` 账本和影响面，不顺手扩大实现范围。

在开始修改前，先建立当前缺口的“改前证据”。优先顺序如下：

1. 能稳定复现该路径问题的自动化测试
2. 能直接证明该路径失败的最小脚本、接口调用、日志或静态对账
3. 如果确实无法自动化，明确记录为什么当前只能做人工或静态验证

不要先写代码、再回头寻找证明。先证明当前路径确实有缺口，再动手修。

如果修 `M1` 的过程中发现必须顺带改 `M2` 才能编译、测试通过或链路成立，先停下说明依赖关系，再把 `M2` 单独登记，不把多个修复点混成一次大改。

## 3. 单点修复循环

每个修复点都必须独立走完以下循环：

`选定 1 个修复点 → 绑定路径 ID → 建立改前证据 → 定义成功信号 → 做最小修改 → 用同一证据回验 → 复查相邻链路 → 记录结果`

展开要求：

1. 一次只允许一个 `in_progress` 的修复点
2. 只处理当前修复点覆盖的缺口，不顺手扩改
3. 改前证据必须能够说明“现在为什么还没修好”；如果证据本身不成立，先修正验证方式，不进入生产代码修改
4. 修改后优先使用同一份改前证据回验，确认缺口真的被关闭，而不是只跑一个更宽泛的检查
5. 修改后立即执行该点验证，不累计多个修复点后再统一验证
6. 当前修复点验证通过后，立刻复查上游入口、下游结果和共用公共方法
7. 当前修复点已验证且相邻链路已复查，才能切到下一个修复点

## 4. 验证优先级

验证顺序固定：

1. **改前证据成立：** 证明当前缺口真实存在
2. **修复点验证：** 用同一证据证明当前修复点成立
3. **相邻链路复查：** 检查同入口不同分支、同公共方法不同入口、同副作用不同触发器
4. **批次级对账：** 更新路径覆盖、未覆盖项和残留风险

不要跳过前两层，直接用一次笼统的“整体 review”代替。

## 5. 改后 review 与测试清单

每个修复点完成验证后，必须立刻做一次面向业务闭环的 review，不能只检查代码能否编译或局部用例是否通过。

### Diff review 闸门

真实 review 必须同时具备以下材料：

- 本次实际改动的 `diff`、修改文件清单或等价代码证据
- 已确认的 `P/M/T` 账本
- 当前修复点绑定的 `M` 编号
- 已执行或无法执行的验证证据

如果缺少以上任一材料，先输出“无法完成真实 review”，并列出缺少什么；不得用经验判断代替 diff 审查，也不得输出“注意权限、注意空值”这类没有代码位置的泛泛提醒。

每条 review 风险都必须包含：

- 文件路径、类名或方法名
- 具体风险点
- 影响的 `P` 路径、`M` 修改点或 `T` 测试项
- 风险等级
- 建议处理方式

没有发现明确问题的维度，写“未发现明确风险”；不要为了凑清单编造风险。

### Review 维度

逐项检查并记录结论：

- **需求覆盖：** 当前修改是否覆盖绑定的 `P` 路径和 `M` 缺口，是否遗漏用户明确提出的输入、输出、页面表现、接口结果或状态变化。
- **入口遗漏：** 是否还有 Controller、内部接口、定时任务、MQ、Webhook、批量任务、管理工具或公共方法能够绕过本次修改。
- **权限与数据归属：** 是否保持原有鉴权、菜单权限、租户/组织/用户归属、数据范围和越权拦截逻辑。
- **状态与幂等：** `status`、`type`、`flag`、`source` 等字段的取值范围和业务含义是否清楚；重复提交、重试、并发和部分失败是否会造成脏状态。
- **异常与空值：** 参数缺失、空集合、空对象、外部接口失败、数据库更新 0 行、事务回滚和异常吞掉时是否有明确处理。
- **ID 精度：** 前端业务 ID、外键 ID、树节点 value 和路由 ID 参数是否保持 `string`，没有把雪花 ID 转成 `number`。
- **性能与范围：** 新增查询、循环、批量处理、聚合明细和公共方法修改是否存在明显性能风险或意外扩大影响面。

### 测试清单

最终输出必须给用户一份可执行的测试清单，每个测试项绑定到 `T` 编号：

| ID | 覆盖路径 | 前置数据/条件 | 操作步骤 | 预期结果 | 验证方式 | 当前状态 |
|---|---|---|---|---|---|---|
| T1 | P1/M1 | … | … | … | 自动化/接口/页面/日志/静态对账 | 已验证/未验证/需人工 |

测试清单至少覆盖：

- 正常主路径
- 本次修复的边界路径
- 被认为不会受影响的相邻路径
- 权限、空值、异常和重复执行场景中与本次修改直接相关的部分

无法验证时，写清阻塞原因，例如缺少依赖服务、缺少测试数据、没有浏览器登录态、无权限访问环境或当前仓库无法启动；不得把“建议验证”写成“已验证”。

## 6. 必须暂停回账的情况

出现以下任一情况时，停止继续编码，先回到路径与影响面分析：

- 当前修改暴露出新的业务入口或绕过路径，且原账本中没有编号
- 为了修 `M1`，开始连续修改多个原本不在计划内的公共点
- 当前验证失败，但失败原因已超出 `M1` 原始缺口
- 连续两次修复都冒出链路外 bug

此时先更新 `P`、`M`、`T` 账本，再决定是否继续。

## 7. 实施类输出要求

如果任务包含“请直接改”“帮我修”“补上这条链路”“落代码”，最终汇报除分析结果外，还必须写清：

- **当前批次：** 本轮只处理哪些 `M`
- **当前修复点：** 正在做哪个 `M`
- **验证方式：** 用什么证据证明这个 `M` 修好
- **复查范围：** 本次改动顺手复查了哪些 `P`
- **Review 结论：** 需求覆盖、入口遗漏、权限、异常、空值、状态、ID 精度和性能风险的检查结果
- **测试清单：** 每个 `T` 的输入、操作、预期、验证方式和当前状态
- **剩余未动项：** 哪些 `M` 仍未开始
- **本轮不处理：** 明确列出本轮故意不处理的 `P`、`M`、模块或旁路

如果当前回合只完成了 `M1`，就明确说只完成了 `M1`，不要把 `M2`、`M3` 写成“应该也已覆盖”。

## 8. 完成前对账

在声称“已修复”之前，必须再次确认：

- 每个已完成的 `M` 都有对应验证证据
- 每条声称已覆盖的 `P` 都能指向至少一个已验证的 `M`
- 本轮没有未登记却已被修改的公共入口、公共方法或旁路逻辑
- 改后 review 已完成，且发现的新风险已经登记为 `U`、`P`、`M` 或测试清单中的未验证项
- 未完成项、未验证边界和建议后续动作仍然保留

没有证据时，就报告真实状态，不要用“整体已完成”掩盖残留问题。
