# 分析细则

本文件负责三件事：

1. 让路径搜索尽量少漏
2. 让分析结果可对账
3. 让后续实施建立在收敛过的路径账本上

## 1. 证据优先级

优先使用当前可获得的最强证据：

1. **代码证据：** 生产代码、SQL、配置、调用点、测试、日志、Git diff
2. **材料证据：** 需求文档、API 文档、流程图、截图、配置、用户明确描述
3. **经验推演：** 证据不足时提出的合理可能性

统一使用以下结论等级：

- **已确认：** 现有证据覆盖了与结论相关的完整链路
- **部分确认：** 有相关证据，但入口、条件、调用关系、持久化结果或外部行为仍有缺口
- **经验推演：** 只有经验支持，没有当前系统直接证据
- **已排除：** 有证据证明在当前检查范围内无法到达目标

没有证据时，不得把合理猜测写成现有路径。

当不同证据互相冲突时，按以下顺序处理：

1. **当前代码与直接运行证据优先：** 先把代码现状、实际日志、实际返回结果、实际持久化结果当作当前系统事实
2. **文档与口述次之：** 需求文档、流程图、历史说明、用户口述可作为目标或期望，但不覆盖当前系统事实
3. **无法统一时记为边界：** 不强行选边，直接登记为 `U`，并写明还缺什么证据

常见冲突处理：

- 代码与文档冲突：默认按代码现状分析，同时说明文档待核对
- 日志与代码预期冲突：优先补运行时边界，检查是否存在隐藏配置、异步、版本差异或外部调用
- 用户描述与当前代码冲突：先说明冲突，再问最小必要问题，确认目标是“当前行为”还是“期望行为”

## 2. 分析流程

### Step 1. 定义目标终点

先整理：

- 目标对象或业务流程
- 起始条件
- 最终状态、行为或副作用
- 成功标准
- 完整性约束，例如“所有”“任何情况”“只执行一次”“最终一定完成”
- 分析范围和明确排除项

如果目标结果、范围边界或成功标准不清楚，并且会改变候选路径集合，先问最小必要问题；否则声明一个范围最小的合理假设并继续。不要因为一般性不确定就停工。

### Step 2. 建立目标搜索锚点

根据实际情况检查目标在系统中的可能表示：

- 枚举名称及数据库数值
- 表名和字段名
- Entity、DTO、VO 属性
- Setter、状态流转、持久化和通知方法
- SQL 赋值、UpdateWrapper、Repository 更新和通用 Patch 接口
- Event、Topic、Scheduled、Webhook、RPC 接口、日志关键字、配置项

同时搜索语义相同的表示和直接写死的值。一个状态可能经由枚举、数字、公共更新方法、整体保存或直接 SQL 写入。

如果材料描述与代码证据相冲突，先把冲突记为验证边界，不强行选一边当事实，并明确说明还需要什么证据才能消除冲突。

### Step 3. 定位所有目标产生点

检查所有可能直接产生目标结果的写入或副作用位置：

- Controller 和内部接口
- Service、状态机、审批流程、公共方法
- Mapper、XML SQL、Repository、批量更新和导入功能
- Scheduled Task、启动任务、数据修复工具、管理接口
- MQ Consumer、Event Listener、Async Task、Webhook、Feign/RPC
- 通用 Entity 保存和字段更新工具

测试代码只能作为行为线索，不能计入生产路径。

### Step 4. 从目标产生点反向追踪入口

对每个目标产生点向上追踪，直到业务入口或系统触发器。确认路径通常应形成：

`入口或触发器 → 业务分支 → 公共调用 → 目标写入或目标行为`

每条路径至少记录：

- 入口或触发器
- 调用链
- 分支条件和状态校验
- 事务、异步、重试、异常处理
- 真正建立目标结果的操作
- 文件、类、方法、SQL 或材料位置

静态代码无法确认动态代理、反射、配置路由、外部调用方或运行时参数时，记为验证边界，不自行猜测。

### Step 5. 从入口类型正向复核到目标

完成反向追踪后，再从候选入口面正向检查是否可能到达目标，不允许只做单向搜索。至少复核：

- Controller、内部 API、管理后台入口
- Scheduled、Job Handler、启动任务
- MQ Consumer、Event Listener、Async Task
- Webhook、Feign/RPC、第三方回调
- 批量任务、导入导出、数据修复工具、脚本入口
- 共享公共方法、通用更新方法、直接 SQL 写入点

正向复核的目标不是重复记录主路径，而是发现：

- 绕过已知目标产生点的旁路
- 入口不同但汇入同一底层实现的独立业务路径

### Step 6. 搜索面对账与排除举证

对每个搜索面给出状态：

- **已检查并命中**
- **已检查未命中**
- **暂未验证**
- **范围外排除**

“已检查未命中”和“范围外排除”都必须附带依据，例如检查过哪些注解、配置、命名模式、字段、目录、日志关键字或调用入口。

没有直接证据时，不要把“未命中”写成“绝对不存在”。

### Step 7. 去重但保留业务差异

只有入口、执行条件、调用链和副作用都相同时，才允许合并路径。

以下差异必须保留为不同路径：

- 直接路径与间接路径
- 单条操作与批量操作
- 同步路径与异步路径
- 正常业务与修复、补偿路径
- 统一状态流转与绕过统一入口的直接写入

### Step 8. 判断需求覆盖与缺口

逐条检查已确认和部分确认路径：

- 是否真的能到达目标
- 用户要求的附加动作是否执行
- 动作发生在持久化或事务提交之前还是之后
- 更新行数为零、回滚、异常吞掉、批量部分成功、重试、并发、重复执行是否会造成“表面成功”
- 是否绕过计划修改的公共位置
- 当前还缺什么

完成日志、方法返回正常、调用 Setter 或尝试执行 SQL，都不能单独证明目标已真正持久化或生效。

### Step 9. 判定搜索是否饱和

只有满足以下条件，才可以说“当前范围内已基本找全候选路径”：

- 至少完成一轮反向追踪
- 至少完成一轮正向复核
- 至少完成一轮旁路检查
- 最近两轮新增搜索不再产生新的 `P` 路径，或新增项只会进入 `U`
- 所有搜索面都已落入四种状态之一

如果条件未满足，就继续查，不因为主链路闭合就提前结束。

这里的“一轮搜索”必须可对账，至少记录：

- 本轮新检查了哪些搜索面、搜索词、注解、目录、配置或入口类型
- 本轮新发现了哪些 `P` 路径
- 本轮新增加了哪些 `U` 未验证边界
- 本轮把哪些原先未确定项转成了“已排除”或“已确认”

只有连续两轮都没有新增 `P` 路径，且新增内容只落入边界补充或排除举证时，才可以说搜索趋于饱和。不能只写一句“已完成两轮搜索”。

## 3. 漏检控制

当任务目标包含“找全路径”“还有哪些入口”“有没有遗漏”“会不会漏掉旁路”等完整性要求时，必须逐项对账以下搜索面：

- Controller、内部 API、管理后台入口
- Service、状态机、审批流、公共业务方法
- Mapper、XML SQL、Repository、UpdateWrapper、直接 SQL
- Scheduled、Job Handler、启动任务
- MQ Consumer、Event Listener、Async Task
- Webhook、Feign/RPC、第三方回调
- 批量任务、导入导出、补偿任务、修复脚本、管理工具
- 配置驱动入口、枚举映射、常量值、日志关键字、消息 Topic

如果项目存在其它明显入口面，也要补进对账表。

### 排除规则

对任何被排除的路径类型，都要说明：

- 检查了哪些代码位置、注解、配置、命名模式或目录
- 为什么这些证据支持当前排除
- 该排除是强排除还是弱排除

其中：

- **强排除：** 有直接证据证明该面无法到达目标
- **弱排除：** 已搜索但未命中，未来仍可能被隐藏配置、外部系统或缺失材料推翻

没有直接证据时，默认按弱排除处理。

## 4. 输出契约

分析结果必须是“可对账的路径清单”，而不是只给结论摘要。即使只找到一条路径，也按此结构输出。

### 1. 目标定义

列出目标结果、成功条件、已检查范围、排除项、必要假设，以及本轮需要对账的搜索面范围。

### 2. 搜索面对账

推荐表格：

| 搜索面 | 检查状态 | 证据位置 | 与目标关系 | 备注 |
|---|---|---|---|---|
| Scheduled Task | 已检查未命中 / 已检查并命中 / 暂未验证 / 范围外排除 | … | … | … |

未检查就写“暂未验证”，不要省略。

### 3. 结论与数量对账

先给一句核心结论，再使用：

`N 条候选路径 = A 条已确认 + B 条部分确认 + C 条经验推演 + D 条已排除`

并列出：需求已覆盖数、未覆盖数、绕过公共位置数、未验证边界数。

### 4. 路径总表

| ID | 路径名称 | 证据等级 | 发现来源 | 入口/触发器 | 目标产生点 | 当前结果 | 需求覆盖 | 是否绕过公共位置 |
|---|---|---|---|---|---|---|---|---|
| P1 | … | 已确认/部分确认/经验推演/已排除 | 反向命中/正向命中/双向一致 | … | … | … | 已覆盖/未覆盖/不适用 | 是/否/未验证 |

分类名称不能代替单条路径。人工入口、定时任务、MQ 消费即使共用底层方法，也分别列成不同路径。

### 5. 逐路径明细

按 `P1` 到 `PN` 展开：

#### P1 — 路径名称【证据等级】

- **发现来源：**
- **入口/触发器：**
- **完整链路：**
- **执行条件：**
- **事务/异步/重试/异常：**
- **当前结果：**
- **需求覆盖与缺口：**
- **证据位置：**
- **所需修改：**

没有证据的字段写“未验证”并说明缺什么。无需修改时写“无—当前已覆盖”或“无—该路径已排除”。

### 6. 未验证边界与经验推演

给每个边界或推演编号 `U1`、`U2` ……，列出：

- 未知内容
- 当前为何无法确认
- 对结论的影响
- 需要什么材料才能确认或排除
- 下一步最小验证动作

### 7. 修改建议与影响范围

给每个修改点编号 `M1`、`M2` ……，列出：

- 修改位置
- 覆盖的路径 ID
- 未覆盖的路径 ID
- 受影响调用方和模块
- 事务、兼容性、发布和监控风险

### 8. 验证计划

给每个验证场景编号 `T1`、`T2` ……，列出：

- 覆盖的路径 ID
- 前置条件
- 操作
- 预期结果
- 要验证的失败边界

### 9. 输出前对账

在给出最终答案前确认：

- 搜索面对账每一行都已标记状态
- `P1` 到 `PN` 连续且无重复
- 路径总表与逐路径明细数量一致
- 已排除项都写明依据
- 没有只在摘要中出现、却没进入路径总表的路径
