# Bug 修复流程模板

本文件用于已有项目中的 bug 修复、线上问题、页面异常、接口异常、数据异常、权限异常、性能异常、定时任务或消息消费异常。核心是先建立复现路径和改前证据，再定位入口、确认根因、生成 `P/M/T` 账本，最后单点修复和回归验证。

## 使用规则

- 第 1 到第 3 段：只排查，不改代码。
- 第 4 段：只修一个明确的 `M` 修复点。
- 第 5 段：基于实际 `diff`、复现证据和 `P/M/T` 做 review，并输出回归测试清单。
- 没有证据的地方写“未确认”，不要猜成事实。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把五段一次性全部执行。

## 第 1 段：复现与现象确认

适用时机：用户刚提供 bug、报错、异常现象、接口返回、日志或线上问题，还没有确认复现路径和改前证据。

```text
请使用 $goal-path-analysis。
先不要改代码。

我遇到一个 bug：
【粘贴 bug 描述 / 报错信息 / 页面现象 / 接口返回 / 日志】

请先帮我整理：

1. Bug 现象
- 当前错误表现是什么？
- 是页面异常、接口异常、数据异常、权限异常、性能异常，还是任务/消息异常？
- 是否稳定复现，还是偶现？

2. 复现路径
- 从哪个页面、接口、任务、消息或操作入口触发？
- 复现步骤是什么？
- 需要什么前置数据或账号权限？
- 如果现在无法复现，还缺什么信息？

3. 期望行为
- 正确情况下页面应该怎样？
- 接口应该返回什么？
- 数据库字段或状态应该怎样？
- 日志、消息、任务执行结果应该怎样？

4. 改前证据
- 哪些页面现象、接口返回、日志、数据库数据或代码位置能证明 bug 存在？
- 哪些证据还缺失？

5. 需要确认的问题
- 哪些信息不确认会影响排查方向？
- 哪些可以先按最小范围假设继续？

只排查，不要改代码。
没有证据的地方请写“未确认”，不要猜成事实。

如果无法稳定复现，请先不要定位代码，先列出需要补充的日志、数据、账号、时间范围或操作视频。
```

合格输出必须让用户看清：错误现象、复现路径、期望行为、改前证据、缺失证据和最小待确认问题。

## 第 2 段：定位入口与调用链

适用时机：复现路径或异常现象已经初步明确，需要从代码证据定位触发入口、调用链、数据和旁路。

```text
请继续使用 $goal-path-analysis。
请基于当前代码定位 bug 入口和调用链，仍然只排查，不要改代码。

请先列出：
1. 你搜索过的关键词
2. 你检查过的文件
3. 你命中的类名、方法名、配置、SQL 或日志关键字

然后按这个格式输出：

1. 触发入口
- 页面入口
- 接口入口
- 定时任务入口
- MQ / Event / Webhook 入口
- 批量任务 / 导入导出 / 后台管理入口

2. 调用链
- Controller
- Service
- Mapper / Repository
- 外部接口 / RPC
- 公共方法
- 异步任务

3. 相关数据
- 表和字段
- 状态、枚举、type、flag、source
- 入参和出参
- 日志关键字

4. 可能旁路
- 是否有其他入口会触发同样逻辑？
- 是否有绕过当前入口的公共方法、SQL、任务或消息？

5. 初步可疑点
- 哪些位置最可能导致 bug？
- 为什么？
- 证据是什么？

每个结论都必须带证据位置：文件路径 + 类名/方法名/SQL/配置。
没有证据的地方请写“未确认”，不要猜成事实。
```

合格输出必须把触发入口、调用链、相关数据、可疑点和旁路都落到证据位置；未查到的面必须标记“未确认”或“已检查未命中”。

## 第 3 段：根因分析 + P/M/T 账本

适用时机：已经有复现证据和调用链，需要判断根因并整理修复账本。

```text
请继续使用 $goal-path-analysis。
请基于复现证据和代码调用链做根因分析，并整理 P/M/T 账本。
先不要改代码。

请输出：

1. 根因判断
- 最可能根因是什么？
- 证据是什么？
- 还有哪些候选根因？
- 哪些候选根因已排除，为什么？
- 哪些仍未确认？

2. P = Bug 触发路径
- 每条 P 从哪里进入
- 调用链是什么
- 哪个条件下触发 bug
- 最终错误结果是什么

3. M = 修复点
- 每个 M 要修什么
- 绑定哪些 P
- 允许修改哪些文件或模块
- 不允许修改哪些文件或模块
- 风险是什么

4. T = 验证项
- 如何复现改前 bug
- 修复后怎么证明问题消失
- 用什么输入数据
- 预期结果是什么
- 验证位置是页面、接口、数据库还是日志

最后请给出推荐修复顺序。
没有我的确认，不要开始改代码。

最后请自检：
1. 哪些根因还只是推测？
2. 哪些 P 没有复现证据？
3. 哪些 M 没有绑定明确 P？
4. 哪些 T 不能证明 bug 被修好？

如果没有足够证据确认根因，请不要进入修复阶段，先说明还需要什么证据。
```

合格输出必须区分最可能根因、候选根因、已排除根因和未确认根因；没有足够证据时不得进入修复。

## 第 4 段：单点修复

适用时机：用户确认 `P/M/T` 账本后，只允许修复一个明确的 `M`。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只修复 M【替换成真实编号】，不要处理其他 M。
例如：现在只修复 M1，不要处理 M2、M3。

动手前请先复述：
1. 本次只修什么
2. 对应的 bug 根因是什么
3. 绑定哪些 P 触发路径
4. 覆盖哪些 T 验证项
5. 允许修改哪些文件
6. 不允许修改哪些文件
7. 本次明确不处理什么

然后开始修改。

改完后请说明：
1. 改了哪个文件
2. 改了什么
3. 为什么改这里
4. 如何对应根因
5. 覆盖哪些 P/M/T
6. 还有哪些路径、根因或修改点没处理
7. 已执行了哪些验证，哪些还没验证
```

合格输出必须明确本轮只修复哪个 `M`，并说明该修改如何对应根因；不得把未处理路径、候选根因或未验证项说成已解决。

## 第 5 段：diff review + 回归测试清单

适用时机：某个 `M` 已修复完成，需要基于真实改动审查并输出回归验证清单。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff、复现证据和 P/M/T 账本严格 review 这次修复，只列有证据的真实风险。

如果缺少 diff、复现证据、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名 / SQL / 日志
影响范围：影响哪个 P/M/T
风险等级：高 / 中 / 低
建议处理：

检查这些方面：
1. 是否真正修复已复现 bug
2. 是否遗漏其他触发入口或旁路
3. 是否只处理了指定 M，没有扩大范围
4. 是否有权限问题
5. 是否有空值、异常、重复提交问题
6. 是否有状态流转问题
7. 是否有 ID 精度问题，尤其前端 ID 不能转 number
8. 是否影响公共方法或其他业务
9. 是否有明显性能风险
10. 是否需要补充回归测试

然后输出回归测试清单：

测试项：
覆盖的 P/M/T：
前置条件：
操作步骤：
预期结果：
验证位置：页面 / 接口 / 数据库 / 日志
是否必须测试：是 / 否
当前状态：已验证 / 未验证 / 需人工确认

要求：
1. 必须包含原 bug 复现路径
2. 必须包含修复后同一路径验证
3. 必须覆盖相关旁路入口
4. 必须覆盖边界场景、异常场景、权限场景
5. 没有证据的测试项标记为“需人工确认”
6. 不要把“建议测试”写成“已验证”
```

合格输出必须证明“原 bug 路径已经回验”，并把未验证、需人工确认和残留风险保留下来。

## 口诀

```text
先复现
找入口
定根因
列 P/M/T
单点修
diff review
回归验证
```
