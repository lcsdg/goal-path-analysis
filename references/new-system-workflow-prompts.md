# 从 0 新系统开发流程模板

本文件用于从 0 开发新系统、新工具、新后台、新页面应用或没有现成代码入口的业务应用。核心是先从业务目标设计路径，再落到模块、页面、接口、数据、技术方案和 `P/M/T` 开发账本。

## 使用规则

- 第 1 到第 5 段：只分析和设计，不写代码。
- 第 6 段：只开发一个明确的 `M` 实现点。
- 第 7 段：必须基于实际 `diff` 和 `P/M/T` 做 review，并输出测试清单。
- 没有依据的内容写“设计假设”；不确定的内容写“待确认”。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把七段一次性全部执行。

## 第 1 段：需求澄清

适用时机：用户刚提出从 0 开发新系统，还没有确认系统目标、角色、MVP 和约束。

```text
请使用 $goal-path-analysis。
先不要写代码。

我要从 0 开发一个新系统：
【粘贴原始需求】

请先帮我整理：
1. 系统目标：这个系统解决什么问题？
2. 用户角色：有哪些用户？各自能做什么？
3. 核心业务流程：用户从进入系统到完成目标的流程是什么？
4. MVP 范围：第一版最小可用版本必须做什么？
5. 暂不处理：哪些功能先不做？
6. 验收标准：什么情况算第一版完成？
7. 需要确认的问题：哪些问题不确认会影响后续设计？
8. 已知限制：技术栈、数据库、部署方式、工期、UI 风格、公司规范有哪些？

只分析，不要写代码。
没有依据的地方请写“设计假设”，不要当成事实。
```

合格输出必须让用户看清：系统目标、角色、MVP、第一版验收标准、设计假设和待确认项。

## 第 2 段：业务路径设计

适用时机：用户认可需求澄清后，需要把系统拆成可编号、可验收的业务路径。

```text
请继续使用 $goal-path-analysis。
继续分析，不要写代码。

请把系统拆成 P 业务路径。

每条 P 按这个格式输出：
P编号：
路径名称：
触发角色：
输入：
业务步骤：
输出结果：
异常情况：
权限要求：
验收标准：

请覆盖主路径、异常路径、权限路径和后台管理路径。
```

合格输出必须覆盖用户完成目标的主路径，并补足异常、权限和后台管理路径；缺少信息时标记“待确认”。

## 第 3 段：模块、页面、接口、数据设计

适用时机：业务路径明确后，需要把路径映射到系统结构。

```text
请继续使用 $goal-path-analysis。
请基于 P 业务路径设计系统结构，仍然不要写代码。

请输出：
1. 前端页面列表
2. 后端接口列表：路径、方法、入参、出参、错误情况
3. 核心数据表或实体
4. 字段含义，尤其 status/type/source/flag 的取值和业务含义
5. 权限规则
6. 状态流转
7. 哪些是 MVP 必须做，哪些可以后做

如果信息不足，请标记“待确认”。
```

合格输出必须把每个关键页面、接口、实体、状态和权限规则关联到至少一条 `P` 路径。

## 第 4 段：技术方案确认

适用时机：模块和数据结构已经初步明确，需要选择技术栈、分层和运行方式。

```text
请继续使用 $goal-path-analysis。
请给出技术实现方案，仍然不要写代码。

请说明：
1. 推荐技术栈
2. 前端目录和组件划分
3. 后端分层：Controller / Service / Repository / DTO
4. 数据库方案
5. 鉴权和权限控制方案
6. 错误处理方案
7. 本地运行方式
8. 主要风险
9. 需要我确认的技术选择
10. 不确定的内容请标记为“待确认”，不要直接拍板。

请给 2-3 个方案，并推荐一个。
```

合格输出必须说明推荐方案的取舍依据，并把未确认的技术限制、运行环境和外部依赖列出来。

## 第 5 段：生成 P/M/T 账本

适用时机：业务路径、系统结构和技术方案都已确认，需要转成开发顺序。

```text
请继续使用 $goal-path-analysis。
请把方案整理成 P/M/T 账本，先不要写代码。

P = 业务路径
M = 开发模块或实现点
T = 验收测试项

每个 M 必须说明：
1. 要实现什么
2. 绑定哪些 P
3. 允许新增或修改哪些文件
4. 不允许做什么
5. 对应哪些 T
6. 风险是什么

最后给出推荐开发顺序。
没有我的确认，不要开始写代码。
```

合格输出必须让用户知道：第一版从哪个 `M` 开始、它覆盖哪些 `P`、用哪些 `T` 验收、哪些模块后做。

## 第 6 段：单点开发

适用时机：用户确认 `P/M/T` 账本后，只允许开发一个明确的 `M`。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只开发 M【替换成真实编号】，不要开发其他 M。
先实现最小可运行版本，不要一次性做完整系统。

动手前请复述：
1. 本次只做什么
2. 覆盖哪些 P
3. 覆盖哪些 T
4. 允许新增或修改哪些文件
5. 不处理哪些内容

然后开始开发。

完成后请说明：
1. 新增或修改了哪些文件
2. 实现了什么
3. 覆盖哪些 P/M/T
4. 哪些还没做
5. 怎么运行和验证
```

合格输出必须产出当前 `M` 的最小可运行实现，并明确未处理的 `M`、`P` 和 `T`。

## 第 7 段：diff review + 测试清单

适用时机：某个 `M` 已完成，需要审查实际改动并给出验收清单。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff 和 P/M/T 账本做 review，只列有证据的真实风险。

如果缺少 diff、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

检查：
1. 是否满足需求和验收标准
2. 是否遗漏业务路径
3. 是否有权限问题
4. 是否有空值、异常、重复提交问题
5. 是否有状态流转问题
6. 是否有 ID 精度问题，前端 ID 不要转 number
7. 是否影响其他模块
8. 是否有明显性能风险

然后输出手工测试清单：
测试项：
覆盖的 P/M/T：
前置条件：
操作步骤：
预期结果：
验证位置：页面 / 接口 / 数据库 / 日志
当前状态：已验证 / 未验证 / 需人工确认

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名
影响范围：影响哪个 P/M/T
风险等级：高 / 中 / 低
建议处理：
```

合格输出必须基于真实改动审查，不得把设计假设、建议测试或未执行验证写成已完成。

## 口诀

```text
定目标
定 MVP
画业务路径
定模块数据接口
定技术方案
列 P/M/T
单点开发
diff review
按清单验收
```
