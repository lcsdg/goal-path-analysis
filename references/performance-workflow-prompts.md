# 性能优化流程模板

本文件用于页面慢、接口慢、SQL 慢、批处理慢、任务堆积、内存或 CPU 异常、超时、吞吐不足和资源消耗过高等性能问题。核心是先量化慢在哪里，建立改前基准，再定位瓶颈、单点优化，最后用同一指标做前后对比。

## 使用规则

- 第 1 到第 4 段：只分析和建立基准，不改代码。
- 第 5 段：只优化一个明确的 `M` 瓶颈点。
- 第 6 段：基于实际 `diff`、性能基准、压测或运行证据和 `P/M/T` 做 review，并输出验证清单。
- 没有指标、日志、执行计划、监控、Profile 或复现证据时写“未确认”，不要凭感觉优化。
- 性能优化必须同时关注正确性，不能为了变快破坏权限、数据一致性、状态流转或业务结果。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把六段一次性全部执行。

## 第 1 段：性能目标与基准确认

适用时机：用户提出慢、卡、超时、任务跑不完、SQL 慢或资源占用高，还没有量化指标和改前基准。

```text
请使用 $goal-path-analysis。
先不要改代码。

我有一个性能优化任务：
【粘贴慢的现象 / 页面 / 接口 / SQL / 任务 / 日志 / 监控 / 数据量】

请先帮我整理：

1. 性能现象
- 慢在哪里：页面、接口、SQL、批处理、定时任务、消息消费、导出还是外部调用？
- 当前耗时、吞吐、错误率、CPU、内存、IO、数据库连接或队列堆积是多少？
- 是稳定慢、偶现慢、峰值慢，还是数据量增长后慢？

2. 业务路径
- 从哪个页面、接口、任务、消息或操作入口触发？
- 涉及哪些用户角色、权限、数据范围和业务状态？

3. 性能目标
- 目标响应时间、吞吐、资源占用或任务完成时间是多少？
- 什么指标证明优化成功？
- 不能牺牲哪些业务正确性？

4. 改前基准
- 需要哪些样例数据、请求参数、数据量、执行计划、日志、监控或 Profile？
- 哪些证据已经有，哪些还缺？

5. 需要确认的问题
- 哪些信息不确认会影响优化方向？
- 哪些可以先按最小范围假设继续？

只分析，不要改代码。
没有指标或证据的地方请写“未确认”，不要凭感觉判断瓶颈。
如果无法建立改前基准，请先说明缺少什么，不要进入优化。
```

合格输出必须让用户看清：慢的对象、触发路径、量化指标、目标指标、改前基准和缺失证据。

## 第 2 段：入口链路与测量点定位

适用时机：性能目标明确后，需要从代码、SQL、日志和监控定位完整链路与可测量点。

```text
请继续使用 $goal-path-analysis。
请基于当前代码和性能证据定位入口链路与测量点，仍然只分析，不要改代码。

请先列出：
1. 你搜索过的关键词
2. 你检查过的文件、SQL、配置、日志或监控项
3. 你命中的类名、方法名、SQL、索引、配置或日志关键字

然后按这个格式输出：

1. 触发入口
- 页面入口
- 接口入口
- 定时任务入口
- MQ / Event / Webhook 入口
- 批量任务 / 导入导出入口

2. 调用链
- Controller / Handler
- Service
- Mapper / Repository / SQL
- 外部接口 / RPC
- 缓存
- 异步任务

3. 测量点
- 哪些位置能记录耗时？
- 哪些 SQL 能看执行计划？
- 哪些日志、traceId、监控或 Profile 能定位耗时？

4. 数据规模
- 查询数据量
- 循环次数
- 批处理大小
- 分页方式
- 索引命中情况
- 缓存命中情况

每个结论都必须带证据位置：文件路径 + 类名/方法名/SQL/配置/日志。
没有证据的地方请写“未确认”，不要猜成事实。
```

合格输出必须能把性能路径从入口追到主要耗时点，并标出可重复测量的位置。

## 第 3 段：瓶颈归因与候选方案

适用时机：链路和测量点明确后，需要确认最可能瓶颈和候选优化方案。

```text
请继续使用 $goal-path-analysis。
请基于改前基准、调用链、SQL、日志或监控做瓶颈归因，先不要改代码。

请输出：

1. 瓶颈判断
- 最可能瓶颈是什么？
- 证据是什么？
- 是数据库、网络、外部接口、循环、序列化、锁、事务、缓存、渲染、打包体积还是资源限制？

2. 候选瓶颈
- 还有哪些候选瓶颈？
- 哪些已排除，为什么？
- 哪些仍未确认，需要什么证据？

3. 优化方案候选
- 方案 A：最小改动方案
- 方案 B：中等改动方案
- 方案 C：长期治理方案

4. 推荐方案
- 推荐先做哪个？
- 预期改善哪个指标？
- 可能带来什么正确性、稳定性或维护性风险？
- 如何回滚？

只分析，不要改代码。
如果没有足够证据确认瓶颈，请不要进入优化阶段，先说明还需要什么指标或日志。
```

合格输出必须把“瓶颈判断”建立在证据上，并说明推荐方案如何用指标验证。

## 第 4 段：生成 P/M/T 账本

适用时机：瓶颈和推荐方案明确后，需要拆成可验证的优化批次。

```text
请继续使用 $goal-path-analysis。
请把性能优化方案整理成 P/M/T 账本，先不要改代码。

P = 性能路径：
- 每条路径从哪里进入
- 调用链是什么
- 当前慢在哪里
- 目标性能指标是什么
- 必须保持什么业务正确性

M = 优化点：
- 每个 M 要优化什么瓶颈
- 绑定哪些 P
- 允许修改哪些文件或模块
- 不允许修改什么
- 预期改善哪个指标
- 回滚方式是什么
- 风险是什么

T = 性能验证项：
- 改前基准是多少
- 优化后目标是多少
- 使用什么数据量、请求参数或压测方式
- 如何验证业务结果不变
- 验证位置是页面、接口、数据库、日志、监控还是压测报告

最后给出推荐优化顺序。
没有我的确认，不要开始改代码。

最后请自检：
1. 哪些 P 没有改前基准？
2. 哪些 M 没有绑定明确瓶颈？
3. 哪些 T 不能证明性能变好或结果不变？
4. 哪些优化缺少回滚方案？
```

合格输出必须把每个优化点绑定到瓶颈证据、目标指标和正确性验证。

## 第 5 段：单点优化

适用时机：用户确认 `P/M/T` 账本后，只允许优化一个明确的 `M`。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只优化 M【替换成真实编号】，不要处理其他 M。
例如：现在只优化 M1，不要处理 M2、M3。

动手前请先复述：
1. 本次只优化什么瓶颈
2. 改前基准和目标指标是什么
3. 必须保持不变的业务结果是什么
4. 绑定哪些 P 性能路径
5. 覆盖哪些 T 验证项
6. 允许修改哪些文件
7. 不允许修改哪些文件
8. 本次明确不处理什么

然后开始修改。

改完后请说明：
1. 改了哪个文件
2. 改了什么
3. 为什么这样优化
4. 预期改善哪个指标
5. 如何保证业务结果不变
6. 覆盖哪些 P/M/T
7. 已执行了哪些验证，哪些还没验证
```

合格输出必须只处理当前瓶颈点，并保留未优化路径、未验证指标和正确性风险。

## 第 6 段：diff review + 性能验证清单

适用时机：某个 `M` 优化完成后，需要基于真实改动审查风险并输出前后对比验证清单。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff、改前基准、优化后验证证据和 P/M/T 账本严格 review 这次性能优化，只列有证据的真实风险。

如果缺少 diff、改前基准、优化后指标、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名 / SQL / 执行计划 / 日志 / 监控
影响范围：影响哪个 P/M/T
风险等级：高 / 中 / 低
建议处理：

检查这些方面：
1. 是否证明性能指标改善
2. 是否保持业务结果不变
3. 是否遗漏其他慢路径、旁路或批量入口
4. 是否有权限、数据归属、状态流转或事务一致性问题
5. 是否有缓存脏读、缓存穿透、缓存击穿或过期策略问题
6. 是否有分页、排序、索引、锁、并发或批量处理问题
7. 是否有 ID 精度、金额精度或时间格式问题
8. 是否影响公共查询、公共组件、公共 Client 或其他业务
9. 是否有资源占用转移、日志缺失或可观测性不足
10. 是否具备回滚方案

然后输出性能验证清单：

测试项：
覆盖的 P/M/T：
改前基准：
优化后结果：
数据量 / 请求参数：
操作步骤：
预期结果：
业务正确性验证：
验证位置：页面 / 接口 / 数据库 / 日志 / 监控 / 压测报告
是否必须测试：是 / 否
当前状态：已验证 / 未验证 / 需人工确认

要求：
1. 必须包含改前和改后同口径对比
2. 必须覆盖主路径和最大数据量路径
3. 必须覆盖边界、异常、权限和并发相关路径
4. 必须验证业务结果不变
5. 没有证据的测试项标记为“需人工确认”
6. 不要把“预计变快”写成“已验证”
```

合格输出必须让用户看清优化是否真的变快、是否仍正确、还有哪些指标未验证。

## 口诀

```text
先量化
建基线
找瓶颈
定方案
列 P/M/T
单点优
前后对比
```
