# 联调 / 接口对接流程模板

本文件用于前后端联调、内部服务接口对接、第三方 API、支付、ERP、平台接口、Webhook、回调、导入导出协议和跨系统数据同步。核心是先定接口契约、入参出参、错误码、鉴权、幂等、日志和联调数据，再按 `P/M/T` 小步对接和验证。

## 使用规则

- 第 1 到第 4 段：只分析和对齐契约，不改代码。
- 第 5 段：只处理一个明确的 `M` 对接点。
- 第 6 段：基于实际 `diff`、接口契约、联调证据和 `P/M/T` 做 review，并输出联调测试清单。
- 没有接口文档、实际请求、响应、日志或代码证据的地方写“未确认”，不要猜成事实。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把六段一次性全部执行。

## 第 1 段：联调目标与契约澄清

适用时机：用户刚提出联调、接口对接或第三方集成，还没有确认双方接口契约和成功标准。

```text
请使用 $goal-path-analysis。
先不要改代码。

我有一个联调 / 接口对接任务：
【粘贴需求、接口文档、对方说明、报错、请求响应或联调现象】

请先帮我整理：

1. 对接目标
- 这次联调要打通什么业务能力？
- 调用方是谁？被调用方是谁？
- 同步调用、异步回调、批量导入导出还是消息消费？

2. 接口契约
- 接口路径、方法、Content-Type、鉴权方式是什么？
- 入参字段、类型、必填、业务含义是什么？
- 出参字段、类型、业务含义是什么？
- 错误码、错误信息和特殊状态有哪些？

3. 验收标准
- 什么情况算联调成功？
- 页面、接口、数据库、日志或外部系统分别应该看到什么？
- 超时、失败、重复请求、权限失败、参数错误时应该怎样？

4. 已知限制
- 环境、账号、token、白名单、网络、回调地址、测试数据、时间窗口有哪些限制？

5. 需要确认的问题
- 哪些契约不确认会影响实现？
- 哪些可以先按最小范围假设继续？

只分析，不要改代码。
没有证据的地方请写“未确认”，不要猜成事实。
```

合格输出必须让用户看清：调用关系、接口契约、验收标准、环境限制和待确认项。

## 第 2 段：调用链与数据流定位

适用时机：接口契约初步明确后，需要基于当前代码或材料定位调用链、数据流和落库点。

```text
请继续使用 $goal-path-analysis。
请基于当前代码和接口材料定位调用链与数据流，仍然只分析，不要改代码。

请先列出：
1. 你搜索过的关键词
2. 你检查过的文件、接口文档或配置
3. 你命中的类名、方法名、配置项、SQL、日志关键字或接口路径

然后按这个格式输出：

1. 调用入口
- 页面入口
- 后端接口入口
- 定时任务入口
- MQ / Event / Webhook 入口
- 批量任务 / 导入导出入口

2. 调用链
- Controller / API Handler
- Service
- Client / Feign / HTTP SDK
- Mapper / Repository
- 回调处理器
- 异步任务

3. 数据流
- 入参如何转换
- 出参如何解析
- 哪些字段落库或回显
- 哪些状态会变化
- 哪些日志能串起一次请求

4. 风险入口
- 鉴权、权限、签名、token、白名单
- 超时、重试、幂等、重复回调
- 字段类型、枚举映射、时间格式、金额精度、ID 精度

每个结论都必须带证据位置：文件路径 + 类名/方法名/接口文档段落/配置项。
没有证据的地方请写“未确认”，不要猜成事实。
```

合格输出必须能把一次调用从入口追到外部系统、回调、落库或页面结果。

## 第 3 段：契约差异与联调数据

适用时机：调用链明确后，需要对比我方实现和对方契约，准备可复现的联调数据。

```text
请继续使用 $goal-path-analysis。
请对比当前实现与接口契约，并整理联调数据。先不要改代码。

请输出：

1. 契约一致项
- 当前实现和文档一致的路径、方法、字段、状态、错误码有哪些？

2. 契约差异项
- 哪些入参、出参、错误码、状态、鉴权、签名、回调或幂等规则不一致？
- 每个差异的证据位置是什么？
- 差异会导致什么联调失败？

3. 联调数据
- 正常请求样例
- 参数错误样例
- 无权限 / token 失效样例
- 重复请求或重复回调样例
- 外部失败或超时样例

4. 日志与排查点
- 请求唯一标识是什么？
- 哪些日志可以串联调用方、被调用方和回调？
- 失败时先看哪里？

没有证据的地方请写“未确认”。
如果契约冲突，请列为待确认，不要直接按任一方拍板。
```

合格输出必须列出可执行联调样例，并明确契约差异是否会影响本次实现。

## 第 4 段：生成 P/M/T 账本

适用时机：接口契约、调用链和联调数据都已有证据后，需要把对接工作拆成可控账本。

```text
请继续使用 $goal-path-analysis。
请把联调分析整理成 P/M/T 账本，先不要改代码。

P = 对接路径：
- 每条路径从哪里进入
- 调用链是什么
- 外部接口、回调、消息或落库点是什么
- 成功和失败结果是什么

M = 对接修改点：
- 每个 M 要处理什么契约差异或实现缺口
- 绑定哪些 P
- 允许修改哪些文件或模块
- 不允许修改什么
- 风险是什么

T = 联调测试项：
- 用什么请求或回调数据
- 预期接口返回、数据库、日志或外部系统结果是什么
- 验证位置是页面、接口、数据库、日志还是第三方平台

最后给出推荐对接顺序。
没有我的确认，不要开始改代码。

最后请自检：
1. 哪些 P 还没有真实契约或代码证据？
2. 哪些 M 没有绑定明确差异或缺口？
3. 哪些 T 不能真正证明联调成功？
```

合格输出必须说明先处理哪个 `M` 能最小闭环打通一条对接路径。

## 第 5 段：单点对接实现

适用时机：用户确认 `P/M/T` 账本后，只允许处理一个明确的 `M`。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只处理 M【替换成真实编号】，不要处理其他 M。
例如：现在只处理 M1，不要处理 M2、M3。

动手前请先复述：
1. 本次只对接什么
2. 绑定哪些 P 对接路径
3. 覆盖哪些 T 联调测试项
4. 允许修改哪些文件
5. 不允许修改哪些文件
6. 本次明确不处理什么

然后开始修改。

改完后请说明：
1. 改了哪个文件
2. 改了什么
3. 为什么改这里
4. 覆盖哪个接口契约或验收标准
5. 覆盖哪些 P/M/T
6. 还有哪些对接路径或修改点没处理
7. 已执行了哪些验证，哪些还没验证
```

合格输出必须明确本轮只打通哪个对接缺口，并保留未处理的契约差异和联调风险。

## 第 6 段：diff review + 联调测试清单

适用时机：某个 `M` 已对接完成，需要基于真实改动审查并输出联调验收清单。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff、接口契约、联调证据和 P/M/T 账本严格 review 这次对接，只列有证据的真实风险。

如果缺少 diff、接口契约、请求响应样例、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名 / 接口文档 / 请求响应 / 日志
影响范围：影响哪个 P/M/T
风险等级：高 / 中 / 低
建议处理：

检查这些方面：
1. 是否满足接口契约和验收标准
2. 是否遗漏调用入口、回调、消息或批量路径
3. 是否有鉴权、权限、签名、token 或白名单问题
4. 是否有参数校验、空值、枚举、时间格式、金额精度或 ID 精度问题
5. 是否有超时、重试、幂等或重复回调问题
6. 是否有错误码、错误信息或异常吞掉问题
7. 是否影响公共 Client、公共 DTO、公共拦截器或其他业务
8. 是否有明显性能风险或日志缺失

然后输出联调测试清单：

测试项：
覆盖的 P/M/T：
请求 / 回调样例：
前置条件：
操作步骤：
预期结果：
验证位置：页面 / 接口 / 数据库 / 日志 / 第三方平台
是否必须测试：是 / 否
当前状态：已验证 / 未验证 / 需人工确认

要求：
1. 必须覆盖正常请求
2. 必须覆盖参数错误、无权限、token 失效、外部失败或超时
3. 必须覆盖重复请求、重复回调或重试幂等
4. 必须覆盖字段映射、状态映射和错误码
5. 没有证据的测试项标记为“需人工确认”
6. 不要把“建议测试”写成“已验证”
```

合格输出必须能直接指导联调，并明确哪些测试已执行、哪些还需要对方或环境配合。

```text
定契约

```
