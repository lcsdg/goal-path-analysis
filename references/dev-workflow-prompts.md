# 已有项目开发流程模板

本文件用于已有项目里的改功能、修 bug、抽公共逻辑、替换实现、排查问题和优化现有逻辑。核心是从现有代码反查入口、影响面和旁路，再用 `P/M/T` 账本控制修改范围。

## 使用规则

- 第 1 到第 3 段：只分析，不改代码。
- 第 4 段：只处理一个明确的 `M` 修改点。
- 第 5 段：基于实际 `diff` 和 `P/M/T` 做 review。
- 第 6 段：按 `P/M/T` 输出测试清单。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把六段一次性全部执行。

## 第 1 段：开工分析

适用时机：用户刚提出已有项目中的开发任务，还没有确认需求、验收标准和初步影响面。

```text
请使用 $goal-path-analysis。
先不要改代码。

我有一个开发任务：
【粘贴原始需求】

请你先帮我整理：

1. 需求理解
- 这个需求要解决什么问题？
- 当前系统可能是什么行为？
- 期望改成什么行为？
- 本次范围是什么？
- 哪些内容暂时不处理？

2. 验收标准
- 什么情况算做完？
- 页面上应该看到什么？
- 接口应该返回什么？
- 数据库字段或状态应该怎么变化？
- 异常、空值、重复操作、无权限时应该怎样？

3. 影响面初判
- 可能涉及哪些页面？
- 可能涉及哪些接口？
- 可能涉及哪些表、字段、状态、枚举？
- 可能涉及哪些权限？
- 是否可能涉及定时任务、MQ、Webhook、导入导出、批量处理或公共方法？

4. 需要确认的问题
- 哪些信息不确认会影响修改方案？
- 哪些可以先按最小范围假设继续？

只分析，不要改代码。
没有证据的地方请写“未确认”，不要猜成事实。

最后请自检：
这一阶段还有哪些结论没有代码证据？
```

合格输出必须让用户看清：当前行为、期望行为、验收标准、初步影响面、未确认项。

## 第 2 段：基于代码找入口

适用时机：用户认可第一阶段方向后，需要用代码证据定位入口、调用链、字段和旁路。

```text
请继续使用 $goal-path-analysis。
请基于当前代码继续找入口和影响面，仍然只分析，不要改代码。

请先列出：
1. 你搜索过的关键词
2. 你检查过的文件
3. 你命中的类名、方法名或配置

然后按这个格式输出：

1. 相关页面入口
2. 相关接口入口
3. Controller / Service / Mapper 调用链
4. 相关字段、状态、枚举
5. 可能的旁路入口：定时任务、MQ、批量任务、导入导出、后台管理入口、公共方法
6. 初步判断应该改哪里
7. 不建议改哪里，为什么

每个结论都必须带证据位置：文件路径 + 类名/方法名。
没有证据的地方请写“未确认”，不要猜成事实。

最后请自检：
这一阶段还有哪些结论没有代码证据？
```

合格输出必须区分“已确认”“未确认”“已排除”，关键结论必须指向文件、类或方法。

## 第 3 段：生成 P/M/T 账本

适用时机：入口和影响面已有证据后，需要把分析结果变成可控的施工账本。

```text
请继续使用 $goal-path-analysis。
请把刚才的分析整理成 P/M/T 账本，先不要改代码。

P = 路径：
- 每条路径从哪里进入
- 调用链是什么
- 最终到达哪个目标逻辑

M = 修改点：
- 每个修改点要改什么
- 绑定哪些 P 路径
- 允许修改哪些文件或模块
- 不允许修改哪些文件或模块
- 风险是什么

T = 测试项：
- 每个修改点怎么验证
- 用什么输入数据
- 预期结果是什么
- 验证位置是页面、接口、数据库还是日志

最后请给出推荐实施顺序。
没有我的确认，不要开始改代码。

最后请自检：
1. 哪些 P 还只是推测？
2. 哪些 M 没有绑定明确 P？
3. 哪些 T 还不能真正证明修改成功？
```

合格输出必须让用户知道：每个 `M` 从哪些 `P` 推导出来、用哪些 `T` 证明完成、为什么推荐当前顺序。

## 第 4 段：确认后小步修改

适用时机：用户确认 `P/M/T` 账本后，只允许处理一个明确的 `M`。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只处理 M【替换成真实编号】，不要处理其他 M。
例如：现在只处理 M1，不要处理 M2、M3。

动手前请先复述：
1. 本次只改什么
2. 绑定哪些 P 路径
3. 覆盖哪些 T 测试项
4. 允许修改哪些文件
5. 不允许修改哪些文件
6. 本次明确不处理什么

然后开始修改。

改完后请说明：
1. 改了哪个文件
2. 改了什么
3. 为什么改这里
4. 覆盖哪个验收标准
5. 覆盖哪些 P/M/T
6. 还有哪些路径或修改点没处理
7. 已执行了哪些验证，哪些还没验证
```

合格输出必须明确本轮只完成哪个 `M`，不得把未处理路径说成已覆盖。

## 第 5 段：改完 Review

适用时机：某个 `M` 已修改完成，需要根据实际改动做审查。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff 和 P/M/T 账本严格 review 这次改动，只列有证据的真实风险。

如果缺少 diff、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名
影响范围：影响哪个 P/M/T
风险等级：高 / 中 / 低
建议处理：

检查这些方面：
1. 是否满足需求和验收标准
2. 是否遗漏入口或旁路
3. 是否有权限问题
4. 是否有空值、异常、重复提交问题
5. 是否有状态流转问题
6. 是否有 ID 精度问题，尤其前端 ID 不能转 number
7. 是否影响公共方法或其他业务
8. 是否有明显性能风险
9. 还需要我手工测试什么

如果某项没有发现问题，请写“未发现明确风险”，不要泛泛提醒。
```

合格输出必须以真实风险为主；没有证据时应返回缺口，而不是输出通用注意事项。

## 第 6 段：生成测试清单

适用时机：review 完成后，需要给用户一份能照着验收的手工测试清单。

```text
请继续使用 $goal-path-analysis。
请基于本次 P/M/T 账本和实际改动，给我一份手工测试清单。

每个测试项按这个格式输出：

测试项：
覆盖的 P/M/T：
前置条件：
操作步骤：
预期结果：
验证位置：页面 / 接口 / 数据库 / 日志
是否必须测试：是 / 否
当前状态：已验证 / 未验证 / 需人工确认

要求：
1. 必须覆盖本次修改的主路径
2. 必须覆盖本次修改相关的边界场景
3. 必须覆盖可能受影响的相邻路径
4. 必须包含异常、空值、重复操作、权限相关测试
5. 没有证据的测试项标记为“需人工确认”
6. 不要把“建议测试”写成“已验证”
```

合格输出必须能被用户直接拿去验证，并明确哪些已经验证、哪些仍需人工确认。

## 口诀

```text
先分析
找证据
列 P/M/T
单点改
按 diff review
按清单验收
```
