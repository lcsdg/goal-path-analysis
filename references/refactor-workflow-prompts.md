# 重构流程模板

本文件用于已有项目中的代码重构、抽象调整、公共方法提取、模块拆分、命名整理、依赖倒置、重复逻辑收敛、架构边界调整和渐进式迁移。核心是先证明行为基线，再拆分迁移批次，保证行为不变、影响可控、可回滚。

## 使用规则

- 第 1 到第 4 段：只分析和设计，不改代码。
- 第 5 段：只重构一个明确的 `M` 批次。
- 第 6 段：基于实际 `diff`、改前基线和 `P/M/T` 做 review，并输出回归清单。
- 默认目标是行为不变；如果用户要求顺便修行为，必须单独登记为功能变更或 bug 修复，不混在重构里。
- 没有代码、测试、日志、接口返回或数据对账证据的地方写“未确认”，不要猜成事实。
- 每阶段没答清楚，就让 AI 补，不进入下一关。
- 一次只执行一个阶段，不要把六段一次性全部执行。

## 第 1 段：重构目标与边界确认

适用时机：用户提出重构、抽公共逻辑、拆模块、整理结构或优化可维护性，还没有确认行为边界和重构成功标准。

```text
请使用 $goal-path-analysis。
先不要改代码。

我有一个重构任务：
【粘贴重构目标 / 代码问题 / 相关文件 / 期望方向】

请先帮我整理：

1. 重构目标
- 这次重构要解决什么维护性问题？
- 是去重复、抽公共方法、拆模块、降复杂度、统一模型、替换依赖，还是清理历史包袱？

2. 行为边界
- 哪些外部行为必须保持不变？
- 页面表现、接口入参出参、数据库结果、日志、消息、权限是否允许变化？
- 哪些行为如果变化，必须先单独确认？

3. 本次范围
- 允许重构哪些模块、文件、类或方法？
- 明确不处理哪些模块、历史问题或附带优化？

4. 成功标准
- 什么证据证明重构成功？
- 什么证据证明行为没有变化？
- 需要哪些改前基线：测试、接口响应、数据库样例、日志、截图或静态调用链？

5. 需要确认的问题
- 哪些信息不确认会影响重构方案或风险？
- 哪些可以按最小范围假设继续？

只分析，不要改代码。
没有证据的地方请写“未确认”，不要猜成事实。
如果无法建立行为不变的基线，请先说明缺少什么，不要进入重构。
```

合格输出必须让用户看清：重构目标、行为不变边界、允许范围、排除项、改前基线和待确认问题。

## 第 2 段：影响面与行为基线

适用时机：重构目标明确后，需要从代码证据找出所有调用方、数据流和行为基线。

```text
请继续使用 $goal-path-analysis。
请基于当前代码建立影响面和改前行为基线，仍然只分析，不要改代码。

请先列出：
1. 你搜索过的关键词
2. 你检查过的文件
3. 你命中的类名、方法名、接口、SQL、配置或测试

然后按这个格式输出：

1. 调用入口
- 页面入口
- Controller / API 入口
- 定时任务
- MQ / Event / Webhook
- 批量任务 / 导入导出
- 内部公共方法调用方

2. 调用链与数据流
- 上游调用方
- 当前被重构点
- 下游依赖
- 数据库读写
- 外部接口、消息或缓存

3. 行为基线
- 改前输入样例
- 改前输出结果
- 改前数据库结果
- 改前日志或副作用
- 已有测试或可复现手工路径

4. 风险面
- 公共方法影响
- 参数兼容
- 状态流转
- 事务、异步、并发、缓存
- 权限与数据归属
- ID 精度、金额精度、时间格式

每个结论都必须带证据位置：文件路径 + 类名/方法名/SQL/配置/测试。
没有证据的地方请写“未确认”，不要猜成事实。
```

合格输出必须覆盖所有已知调用方，并把“可证明行为不变”的基线列出来。

## 第 3 段：重构方案与迁移策略

适用时机：影响面和基线清楚后，需要设计低风险重构方案。

```text
请继续使用 $goal-path-analysis。
请基于影响面和行为基线设计重构方案，先不要改代码。

请输出：

1. 方案候选
- 方案 A：最小改动方案
- 方案 B：结构更清晰方案
- 方案 C：可选的长期演进方案

2. 推荐方案
- 推荐哪一个？
- 为什么？
- 它如何保证行为不变？
- 它的改动半径是什么？

3. 迁移策略
- 是否需要先包一层适配器？
- 是否需要保留旧入口一段时间？
- 是否需要分批替换调用方？
- 是否需要灰度、开关、回滚方案？

4. 不建议方案
- 哪些重构方式不建议做？
- 为什么风险高或收益低？

5. 待确认项
- 哪些规则、调用方、历史行为还不能确认？

只输出方案，不要改代码。
如果重构会改变业务行为，请单独标记为“行为变更”，不要伪装成重构。
```

合格输出必须给出推荐方案和不推荐方案，并说明如何保证行为不变和如何回滚。

## 第 4 段：生成 P/M/T 账本

适用时机：重构方案确认后，需要拆成可验证的迁移批次。

```text
请继续使用 $goal-path-analysis。
请把重构方案整理成 P/M/T 账本，先不要改代码。

P = 受影响路径：
- 每条路径从哪里进入
- 调用链是什么
- 当前依赖哪个旧实现
- 重构后应保持什么行为

M = 重构批次或修改点：
- 每个 M 要重构什么
- 绑定哪些 P
- 允许修改哪些文件或模块
- 不允许修改什么
- 是否需要兼容层、适配器、开关或回滚点
- 风险是什么

T = 行为不变验证项：
- 用什么改前基线验证
- 重构后预期结果是什么
- 验证位置是测试、页面、接口、数据库还是日志

最后给出推荐迁移顺序。
没有我的确认，不要开始改代码。

最后请自检：
1. 哪些 P 还没有调用方证据？
2. 哪些 M 可能改变行为？
3. 哪些 T 不能证明行为不变？
4. 哪些回滚或兼容策略还不清楚？
```

合格输出必须把每个重构批次绑定到受影响路径和行为不变验证项。

## 第 5 段：单点重构

适用时机：用户确认 `P/M/T` 账本后，只允许执行一个明确的 `M` 重构批次。

```text
请继续使用 $goal-path-analysis。
我确认这个 P/M/T 账本。

现在只重构 M【替换成真实编号】，不要处理其他 M。
例如：现在只重构 M1，不要处理 M2、M3。

动手前请先复述：
1. 本次只重构什么
2. 必须保持不变的行为是什么
3. 绑定哪些 P 路径
4. 覆盖哪些 T 验证项
5. 允许修改哪些文件
6. 不允许修改哪些文件
7. 本次明确不处理什么

然后开始修改。

改完后请说明：
1. 改了哪个文件
2. 改了什么
3. 为什么这样重构
4. 如何保证行为不变
5. 覆盖哪些 P/M/T
6. 还有哪些路径或重构批次没处理
7. 已执行了哪些验证，哪些还没验证
```

合格输出必须只完成当前重构批次，并用改前基线或等价证据说明行为是否保持一致。

## 第 6 段：diff review + 回归清单

适用时机：某个 `M` 重构完成后，需要基于真实改动审查行为风险。

```text
请继续使用 $goal-path-analysis。
请基于本次 diff、改前行为基线和 P/M/T 账本严格 review 这次重构，只列有证据的真实风险。

如果缺少 diff、改前基线、P/M/T 账本或代码证据，请先说明缺少什么，不要凭经验泛泛 review。

每个风险请按这个格式输出：
风险点：
证据位置：文件路径 / 方法名 / SQL / 测试 / 日志
影响范围：影响哪个 P/M/T
风险等级：高 / 中 / 低
建议处理：

检查这些方面：
1. 是否保持行为不变
2. 是否遗漏调用方、旁路或公共方法影响
3. 是否改变接口入参、出参、错误码、状态或数据库结果
4. 是否改变权限、数据归属、事务、异步、缓存或幂等行为
5. 是否有空值、异常、重复提交、并发问题
6. 是否有 ID 精度、金额精度或时间格式问题
7. 是否留下新旧逻辑双写不一致、死代码或兼容层风险
8. 是否有明显性能退化
9. 是否需要补充回滚或灰度方案

然后输出回归测试清单：

测试项：
覆盖的 P/M/T：
改前基线：
操作步骤：
预期结果：
验证位置：测试 / 页面 / 接口 / 数据库 / 日志
是否必须测试：是 / 否
当前状态：已验证 / 未验证 / 需人工确认

要求：
1. 必须覆盖所有受影响主路径
2. 必须覆盖公共方法的主要调用方
3. 必须覆盖边界、异常、权限和并发相关路径
4. 必须包含行为不变对比
5. 没有证据的测试项标记为“需人工确认”
6. 不要把“建议测试”写成“已验证”
```

合格输出必须证明重构前后行为一致，或明确指出哪些路径仍需验证。

