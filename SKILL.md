# 目标全路径分析

把用户期望的结果作为终点，反向寻找当前系统到达该终点的全部可验证路径，判断每条路径是否满足需求，并说明还缺什么、应该修改哪里以及会影响什么。

## 基本原则

- 代码仓库和运行证据的优先级高于方法名称、业务常识和开发经验。
- 没有证据支持的合理路径，不得描述为系统中已经存在的路径。
- 不声称绝对找到了所有路径。必须说明已检查范围以及无法验证的边界。
- 区分业务入口和底层公共实现。多个业务场景即使最终调用同一个方法，也不能直接合并成一条路径。
- 遵守用户要求的操作范围。用户只要求审查或分析时，不修改代码。
- 用户要求实现时，先完成路径与影响面分析，再按照确认的路径实施修改。
- 根据任务风险控制分析深度，不把明显独立的小改动扩大成无必要的架构审查。

## 选择证据模式

优先使用当前可获得的最强证据，必要时组合使用以下模式。

1. **代码证据模式：** 根据生产代码、SQL、配置、调用点、测试、日志和 Git diff 还原路径。每条确认路径必须能够从入口或触发器，经由条件和调用关系，追踪到目标结果。
2. **材料证据模式：** 根据需求文档、API 文档、流程图、日志、截图、配置或用户明确描述还原路径。每条确认路径都要指出其依据。
3. **经验推演模式：** 证据不足时，可以列出技术上常见或合理的路径，但只能作为推演；同时说明推演理由，以及需要什么材料才能确认或排除。

统一使用以下结论等级：

- **已确认：** 现有证据覆盖了与结论相关的完整链路。
- **部分确认：** 存在相关证据，但入口、条件、调用关系、持久化结果或外部行为中仍有环节未验证。
- **经验推演：** 根据开发或业务经验提出，当前系统中没有直接证据。
- **已排除：** 有证据证明在所检查条件下，该路径无法到达目标。

## 分析流程

### 1. 定义目标终点

将用户需求整理成明确的目标契约：

- 目标对象或业务流程
- 起始条件（如适用）
- 最终状态、行为或副作用
- 怎样才算真正完成
- “所有”“任何情况”“只执行一次”“最终一定完成”等完整性约束
- 分析范围和明确排除项

只有在歧义会显著改变排查方向时才询问用户；否则说明一个范围较小的合理假设并继续。

### 2. 建立目标搜索锚点

识别目标在系统中的各种可能表示。代码任务根据实际情况检查：

- 枚举名称及其数据库数值
- 表名和字段名
- Entity、DTO、VO 中的属性
- Setter、状态流转、持久化和通知方法
- SQL 赋值、UpdateWrapper、Repository 更新和通用 Patch 接口
- Event、Topic、Scheduled Task、Webhook、RPC 接口、飞书卡片类型、日志关键词和配置项

同时搜索语义相同的表示和直接写死的数值。一个状态可能通过枚举、数字、公共更新方法、Entity 整体保存或直接 SQL 写入。

### 3. 查找所有目标产生点

定位所有能够产生目标结果的写入、状态转换、消息发送或其他操作。根据项目实际存在的机制检查：

- Controller 和内部接口
- Service、状态机、审批流程和公共方法
- Mapper、XML SQL、Repository、批量更新和导入功能
- Scheduled Task、启动任务、数据修复工具和管理接口
- MQ Consumer、Event Listener、Async Task、Webhook 和 Feign/RPC
- 通用 Entity 保存和字段更新工具
- 测试代码只能作为行为线索，不能计入生产路径

仓库没有暴露数据库触发器、外部任务或人工操作的证据时，不得虚构它们。若这些边界对结论很重要，应标记为无法验证的外部边界。

### 4. 从产生点反向追踪入口

针对每个目标产生点，继续追踪调用方和执行条件，直到到达有业务意义的入口或系统触发器。代码中的确认路径通常应形成：

`入口或触发器 → 业务分支 → 公共调用 → 目标写入或目标行为`

每条路径记录：

- 入口或触发器
- 调用链或流程步骤
- 分支条件和状态校验
- 相关的事务、异步、重试和异常处理
- 真正建立目标结果的操作
- 文件、类、方法、SQL 或材料位置

静态代码无法确认动态代理、反射、配置路由、外部调用方或运行时参数时，标记验证边界，不自行猜测。

### 5. 去重但保留业务差异

合并入口和条件都相同的重复调用链。不同触发方式、权限、条件分支、重试方式或副作用即使最终汇入同一个底层方法，也要保留为不同业务路径。

明确区分：

- 直接路径与间接路径
- 单条操作与批量操作
- 同步路径与异步路径
- 正常业务与修复、补偿路径
- 统一状态流转与绕过统一入口的直接写入

### 6. 判断需求覆盖情况

逐条检查已确认和部分确认的路径：

- 是否真的能够到达目标
- 用户要求的附加动作当前是否执行
- 动作发生在持久化或事务提交之前还是之后
- 更新行数为零、事务回滚、异常被吞、批量部分成功、重试、并发或重复执行是否会导致表面成功但目标未完成
- 是否绕过计划修改的公共位置
- 该路径距离目标还缺少什么

完成日志、方法正常返回、调用 Setter 或尝试执行 SQL，都不能单独证明数据已经持久化。结合影响行数、事务边界、异常传播和可获得的运行证据判断。

### 7. 设计修改方案

推荐能够覆盖全部已确认路径的最小可靠修改。只有当前架构适合且所有相关路径确实经过该位置时，才建议放入统一状态方法、领域事件或公共成功回调。

如果存在绕过公共位置的路径，应将其收口到统一流程，或者明确列出必须单独修改的位置。只分析与当前功能有关的风险，例如：

- 事务时机与回滚
- 幂等和重复状态转换
- 重试与失败处理策略
- 批量部分成功
- 权限和状态校验
- 并发更新
- 外部系统行为
- 历史数据及兼容性
- 日志、监控和更新行数校验

如果最小改动和结构性优化都有价值，分别给出，不强行重构。

### 8. 生成验证场景

测试场景必须来自已确认路径和重要失败边界，不能只是通用检查清单。覆盖每个独立入口、绕过路径、关键条件分支、重复调用、持久化失败或事务回滚，以及相关异步或外部系统失败。

## 输出格式

根据任务复杂度调整篇幅，但复杂分析应尽量保留以下结构。

### 目标定义

说明目标结果、成功条件、分析范围和必要假设。

### 结论摘要

给出核心结论及数量，例如已确认路径、部分确认路径、需求未覆盖路径、绕过入口和经验推演数量。

### 路径清单

每条路径包含：

1. 路径名称和证据等级
2. 入口或触发器
3. 到达目标的调用链或流程
4. 执行条件
5. 当前结果
6. 需求覆盖情况或缺口
7. 证据位置
8. 所需修改

路径较多时先使用紧凑表格汇总，再展开需要解释的路径。

### 未验证边界与经验推演

列出缺失的代码仓库、外部系统、运行配置、动态调用方或基于经验提出的可能性，并说明如何验证每一项。

### 修改建议与影响范围

指出修改位置、每个位置覆盖的路径、需要单独处理的绕过路径、受影响的调用方和模块，以及事务、兼容性、发布和监控风险。

### 验证计划

列出从真实路径推导出的测试场景和预期结果。

## 示例

面对“订单变成已关闭时发送飞书”，不能找到 `closeOrder()` 就结束。应先寻找所有写入“已关闭”状态的位置，再根据项目证据追踪人工关闭、平台取消回调、定时关闭、批量操作、取消流程、修复接口和通用更新等入口。随后说明哪些路径已经通知、哪些路径遗漏、公共修改点能够覆盖哪些路径，以及哪些可能性因为缺少证据只能列为经验推演。
