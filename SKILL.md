# 目标全路径分析

把目标结果当作终点，找出当前系统到达该终点的全部可验证路径，区分事实、排除、未验证边界和经验推演，并在需要落代码时按路径分批实施，减少返工。

## 何时使用

使用这个 skill，当用户要求你：

- 找全某个状态、结果、副作用或行为的所有入口与路径
- 判断“还有没有遗漏”“会不会漏掉旁路”“会影响哪里”
- 先分析链路，再决定修改位置、风险和验证方案
- 按“先分析、找证据、列 P/M/T、单点改、diff review、测试清单”的六阶段流程推进开发任务
- 按“先复现、找入口、定根因、列 P/M/T、单点修、diff review、回归验证”的五阶段流程修复 bug
- 从 0 开发新系统，并需要先定目标、MVP、业务路径、模块接口数据、技术方案和 P/M/T 账本
- 做联调、接口对接、第三方 API、Webhook、支付、ERP 或跨系统同步，并需要先对齐接口契约和联调数据
- 做重构、抽象调整、模块拆分、迁移或公共方法收敛，并需要保证行为不变
- 做性能优化、SQL 优化、接口提速、批处理优化或任务吞吐优化，并需要改前基准和前后对比
- 处理线上故障、告警、紧急问题、资损风险、数据错误、任务堆积或发布事故，并需要先止血、保数据和复盘
- 在多入口系统里修复 bug，并希望减少反复 review 和返工

不要用于：

- 纯概念解释
- 明确只改一个完全局部、不会穿过公共流程的小改动
- 单纯文案、格式或机械替换

## 核心承诺

- 不因为找到主链路就提前收工。
- 不把缺证据的猜测说成已存在路径。
- 不把多个业务入口粗暴合并成一条路径。
- 不在路径账本没收敛前大面积改代码。
- 不在没有验证证据时声称“已修复”。
- 不跳过需求复述、验收标准、影响面、修改计划、review 和验证清单这些 AI 协作开发关口。

## AI 协作开发硬闸门

当任务涉及代码分析、修复、功能实现、联调、review 或测试清单时，先把工作拆成可验收的协作流程，不直接进入“边看边改”：

1. **需求理解：** 用自己的话复述用户目标，区分当前行为、期望行为、范围边界和排除项。
2. **验收标准：** 写清什么现象、接口结果、页面表现、数据状态或副作用能证明任务完成。
3. **影响面扫描：** 列出可能涉及的页面、接口、字段、状态、权限、任务、消息、外部回调和公共方法。
4. **路径账本：** 在改代码前形成 `P` 路径清单；实施任务继续补齐 `M` 修改点和 `T` 验证场景。
5. **修改计划：** 每个 `M` 必须绑定具体 `P` 和 `T`，说明最大改动半径，不把多个缺口混成一次大改。
6. **改后 review：** 修改后按需求覆盖、遗漏入口、权限、异常、空值、并发、状态流转、ID 精度和性能风险逐项复查。
7. **验证交付：** 最终说明已验证项、未验证项、残留风险和建议的手工测试路径。

如果任务是 bug 修复，必须先建立复现路径和改前证据；无法稳定复现时，先列出缺少的日志、数据、账号、时间范围或操作视频，不直接猜根因或改代码。

如果任务是线上故障，必须先判断影响范围、是否仍在扩大、是否需要止血、是否存在资损或数据损坏风险；任何删除数据、重置代码、执行补偿脚本、影响外部系统或扩大线上影响的操作，都必须先获得用户明确确认。

如果任务是性能优化，必须先建立改前基准和目标指标；没有指标、日志、执行计划、监控、Profile 或复现证据时，不得凭感觉优化。

如果任务是重构，默认行为不变；任何业务行为变化都必须单独登记为功能变更或 bug 修复，不得伪装成重构。

如果用户只要求分析、排查、review 或评估影响，停在分析模式；除非用户明确说“开始改”“帮我修”“直接落代码”“补上实现”，否则不得修改文件。

## 执行闸门

出现以下情况时，先停下来，不要硬推进：

- 目标结果、范围边界或成功标准不清楚，而且会改变候选路径集合
- 当前只有猜测，没有足够证据把某条路径写成事实
- 用户没有明确授权改代码，却已经开始滑向实施模式
- 路径账本还没收敛，却已经准备大面积改公共流程
- 修复点还没有验证证据，却想直接声称“已修复”

## 规模闸门

如果出现以下任一情况，不要承诺“一次性做完”，先拆批次、拆路径组或拆子问题：

- 候选路径明显很多，已经跨多个入口面
- 影响范围跨多个模块、子系统或外部集成点
- 需要同时修改多个公共方法、消息链路、定时任务或数据库写入面
- 单次实施无法在一个清晰账本里维持 `P/M/T` 对账

此时先给出拆分方式，再说明当前批次先处理哪一组路径。

## 提问闸门

只有当下面的信息缺失，并且会改变候选路径集合、修改范围或是否允许实施时，才停下来问用户：

- 目标结果定义不清
- 范围边界不清
- 成功标准不清
- 是否授权直接改代码不清

提问时只问最小必要问题，优先 1 到 2 个窄问题。除此之外，先声明一个范围最小的合理假设并继续，不把一般性不确定当成停工理由。

## 先判定开发场景与任务模式

先明确当前开发场景属于哪一种：

1. **线上故障 / 紧急问题：** 用户提到线上、生产、告警、故障、事故、紧急、资损、数据损坏、用户受影响、发布事故、回滚、补偿、任务堆积或影响仍在扩大。
2. **Bug 修复：** 用户提到 bug、报错、异常、不生效、数据不对、页面异常、接口返回不对、偶现问题、定时任务失败或消息消费异常，但没有明确线上故障或紧急止血语境。
3. **联调 / 接口对接：** 用户提到联调、接口对接、API、第三方、支付、ERP、Webhook、回调、签名、token、白名单、错误码、入参出参或跨系统同步。
4. **性能优化：** 用户提到慢、卡、超时、SQL 慢、接口慢、页面慢、批处理慢、吞吐不足、CPU、内存、IO、锁、索引、缓存或资源占用。
5. **重构：** 用户提到重构、抽象调整、模块拆分、迁移、去重复、抽公共方法、整理结构、依赖倒置、行为不变或代码可维护性。
6. **从 0 新系统：** 用户提到从 0 开发、新系统、没有代码、搭建一个系统、新工具、新后台、新页面应用，或当前重点是先设计业务路径、模块、接口、数据和技术方案。
7. **已有项目变更：** 用户提到当前代码、已有系统、改功能、替换实现、排查、优化现有逻辑、联调或基于现有项目继续开发，但不属于以上更具体场景。

场景判定按上面的优先级执行：线上故障优先级最高；联调、性能、重构优先于普通已有项目变更；从 0 新系统优先于当前工作区是否存在文件。如果用户明确说“从 0”“新系统”“没有代码”，即使当前工作区存在文件，也按从 0 新系统处理。

如果用户没有说清楚开发场景，但任务描述包含“线上、生产、告警、故障、资损、数据损坏、用户受影响”，默认按线上故障处理；包含“bug、报错、异常、不生效、数据不对、偶现”，默认按 Bug 修复处理；包含“联调、接口、第三方、Webhook、回调、签名、token、错误码”，默认按联调 / 接口对接处理；包含“慢、超时、性能、SQL、吞吐、CPU、内存、缓存、索引”，默认按性能优化处理；包含“重构、抽公共、拆模块、迁移、行为不变”，默认按重构处理；当前工作区已有可读代码且任务描述是普通“改、替换、排查、优化”，默认按已有项目变更处理。

再明确当前任务模式属于哪一种：

1. **分析/设计模式：** 用户要你找全路径、判断遗漏、评估影响、设计新系统、生成流程/方案/账本，但不要求直接改代码。
2. **实施模式：** 用户要求你基于路径分析继续修复、补链路、落代码、开始开发某个 `M` 或提交修复。

默认先进入分析/设计模式。只有当用户明确表达“直接改”“帮我修”“补上这条链路”“落代码”“开始开发 M1”“提交修复”等实施授权时，才进入实施模式。

如果用户只要求“看一下”“排查”“分析”“review”“评估影响”“找全路径”“设计一下”“列方案”“生成 P/M/T”，一律停在分析/设计模式，不修改代码。

## 线上故障六阶段开发协作映射

当用户处理线上故障、告警或紧急问题时，按以下映射执行：

1. **故障分级与止血判断：** 分析/设计模式。先判断影响范围、是否仍在扩大、是否存在资损、数据损坏或权限泄露风险，并列出需要授权的止血选项。
2. **影响范围与入口路径对账：** 分析/设计模式。用线上证据、日志、监控、配置、数据和代码定位受影响入口、扩散路径、数据影响和外部影响。
3. **根因定位与处置方案：** 分析/设计模式。区分止血、修复、回滚、补偿和数据保护方案；证据不足时不进入修复。
4. **生成 P/M/T 处置账本：** 分析/设计模式。把故障路径、处置点和验证项编号，并标明授权要求、回滚方式和风险。
5. **单点处置：** 实施模式。一次只处理用户指定的一个 `M`，且高风险线上操作必须先获得明确授权。
6. **diff review + 验证补偿复盘清单：** 实施模式。必须基于实际 `diff` 或操作记录、线上证据、影响面和 `P/M/T` review，并输出验证、补偿和复盘清单。

六个阶段是关卡，不是一次性大 prompt。前四个阶段不改代码、不执行线上操作；第五个阶段只处理一个 `M`；第六个阶段不基于证据就不做泛泛 review。

## 联调接口六阶段开发协作映射

当用户做联调、接口对接、第三方 API、Webhook 或跨系统同步时，按以下映射执行：

1. **联调目标与契约澄清：** 分析/设计模式。先对齐调用方、被调用方、接口路径、方法、入参出参、错误码、鉴权和验收标准。
2. **调用链与数据流定位：** 分析/设计模式。用代码、配置、接口文档和日志定位入口、Client、回调、消息、落库和状态变化。
3. **契约差异与联调数据：** 分析/设计模式。对比我方实现和对方契约，整理正常、异常、无权限、重复请求、超时等联调样例。
4. **生成 P/M/T 账本：** 分析/设计模式。把对接路径、契约缺口、实现点和联调测试项编号；没有用户确认不得改代码。
5. **单点对接实现：** 实施模式。一次只处理用户指定的一个 `M`，不顺手改其他契约差异。
6. **diff review + 联调测试清单：** 实施模式。必须基于实际 `diff`、接口契约、请求响应、日志和 `P/M/T` review，并输出联调测试清单。

六个阶段是关卡，不是一次性大 prompt。前四个阶段不写代码；第五个阶段只处理一个 `M`；第六个阶段不基于证据就不做泛泛 review。

## 重构六阶段开发协作映射

当用户做重构、抽象调整、模块拆分、迁移或公共方法收敛时，按以下映射执行：

1. **重构目标与边界确认：** 分析/设计模式。先确认重构目标、允许范围、排除项和必须保持不变的外部行为。
2. **影响面与行为基线：** 分析/设计模式。用代码、测试、接口响应、数据库样例、日志或截图建立改前行为基线和调用方清单。
3. **重构方案与迁移策略：** 分析/设计模式。给出候选方案、推荐方案、兼容层、分批迁移、灰度和回滚策略。
4. **生成 P/M/T 账本：** 分析/设计模式。把受影响路径、重构批次和行为不变验证项编号；没有用户确认不得改代码。
5. **单点重构：** 实施模式。一次只重构用户指定的一个 `M`，默认行为不变。
6. **diff review + 回归清单：** 实施模式。必须基于实际 `diff`、改前基线和 `P/M/T` review，并输出行为不变回归清单。

六个阶段是关卡，不是一次性大 prompt。重构不得夹带业务行为变化；行为变化必须单独登记为功能变更或 bug 修复。

## 性能优化六阶段开发协作映射

当用户做页面、接口、SQL、批处理、任务、消息或资源占用优化时，按以下映射执行：

1. **性能目标与基准确认：** 分析/设计模式。先量化慢在哪里，建立改前指标、目标指标、数据量和正确性边界。
2. **入口链路与测量点定位：** 分析/设计模式。用代码、SQL、日志、监控、执行计划或 Profile 定位完整链路和可测量点。
3. **瓶颈归因与候选方案：** 分析/设计模式。区分最可能瓶颈、候选瓶颈、已排除瓶颈和未确认瓶颈；证据不足时不进入优化。
4. **生成 P/M/T 账本：** 分析/设计模式。把性能路径、优化点和性能验证项编号，并绑定改前基准、目标指标和回滚方式。
5. **单点优化：** 实施模式。一次只优化用户指定的一个 `M`，同时保证业务结果不变。
6. **diff review + 性能验证清单：** 实施模式。必须基于实际 `diff`、改前基准、优化后指标和 `P/M/T` review，并输出同口径前后对比清单。

六个阶段是关卡，不是一次性大 prompt。没有改前基准和优化后同口径对比，不得声称“已优化”。

## 已有项目六阶段开发协作映射

当用户按阶段推进开发任务，或要求把本 skill 配合日常开发流程使用时，按以下映射执行：

1. **开工分析：** 分析模式。输出需求理解、验收标准、影响面初判和需要确认的问题，不读取或修改无关文件。
2. **基于代码找入口：** 分析模式。用代码证据定位页面、接口、调用链、字段、状态、枚举和旁路入口；结论必须带文件、类或方法位置。
3. **生成 P/M/T 账本：** 分析模式。把 `P` 路径、`M` 修改点、`T` 验证项编号，并给出推荐实施顺序；没有用户确认不得改代码。
4. **确认后小步修改：** 实施模式。一次只处理用户指定的一个 `M`；动手前复述改动半径，改完后说明覆盖哪些 `P/M/T`。
5. **改完 review：** 实施模式。必须基于实际 `diff`、`P/M/T` 账本和证据位置审查；缺少这些材料时先报告缺口，不给泛泛风险。
6. **生成测试清单：** 实施模式。基于实际改动和 `P/M/T` 输出可执行测试清单，区分已验证、未验证和需人工确认。

六个阶段是关卡，不是一次性大 prompt。除非用户明确要求整合模板，否则不要把后续阶段提前执行。

## Bug 修复五阶段开发协作映射

当用户修 bug、排查报错或处理异常现象时，按以下映射执行：

1. **复现与现象确认：** 分析/设计模式。先整理错误现象、复现路径、期望行为、改前证据和缺失证据；无法稳定复现时，不直接定位代码。
2. **定位入口与调用链：** 分析/设计模式。用代码证据定位页面、接口、任务、MQ、Webhook、批量入口、调用链、相关数据和旁路。
3. **根因分析 + P/M/T 账本：** 分析/设计模式。区分最可能根因、候选根因、已排除根因和未确认根因；没有足够证据确认根因时，不进入修复。
4. **单点修复：** 实施模式。一次只修复用户指定的一个 `M`，动手前复述根因、绑定路径、验证项和改动半径。
5. **diff review + 回归测试清单：** 实施模式。必须基于实际 `diff`、复现证据、`P/M/T` 账本和代码证据 review，并输出原 bug 路径和相关旁路的回归测试清单。

五个阶段是关卡，不是一次性大 prompt。前三个阶段不改代码；第四个阶段只修一个 `M`；第五个阶段不基于证据就不做泛泛 review。

## 从 0 新系统七阶段开发协作映射

当用户从 0 开发新系统，或没有现成代码入口可反查时，按以下映射执行：

1. **需求澄清：** 分析/设计模式。输出系统目标、用户角色、核心流程、MVP、暂不处理项、验收标准、已知限制和待确认问题。
2. **业务路径设计：** 分析/设计模式。把用户、系统和后台管理动作拆成 `P` 业务路径；没有事实依据时写“设计假设”，不写“已确认”。
3. **模块、页面、接口、数据设计：** 分析/设计模式。把 `P` 映射到页面、接口、实体、字段、状态、权限和状态流转。
4. **技术方案确认：** 分析/设计模式。给出 2 到 3 个技术方案和推荐方案；不确定内容写“待确认”，不直接拍板。
5. **生成 P/M/T 账本：** 分析/设计模式。把 `P` 业务路径、`M` 开发模块或实现点、`T` 验收测试项编号；没有用户确认不得写代码。
6. **单点开发：** 实施模式。一次只开发用户指定的一个 `M`，先做最小可运行版本，不一次性做完整系统。
7. **diff review + 测试清单：** 实施模式。必须基于实际 `diff`、`P/M/T` 账本和代码证据 review，并输出手工测试清单。

七个阶段是关卡，不是一次性大 prompt。前五个阶段不写代码；第六个阶段只开发一个 `M`；第七个阶段不基于证据就不做泛泛 review。

## 最小工作流

### 1. 先定义目标

先写清：

- 目标对象或流程
- 最终状态、结果或副作用
- 成功标准
- 本轮范围和排除项
- 当前行为和期望行为
- 可能涉及的页面、接口、字段、状态、权限和外部触发面

只有当以下任一信息不清楚，并且会改变候选路径集合时，才停下来问用户：

- 目标结果是什么
- 范围边界是什么
- 什么算真正完成

否则，声明一个范围最小的合理假设并继续。

### 2. 先收敛路径，再决定改哪里

在提出修改方案或开始编码前，先形成最小账本：

- 候选路径 `P1...PN`
- 未验证边界 `U1...UN`
- 如果要实施，再补 `M1...MN` 和 `T1...TN`

没有路径账本时，不进入大范围修改。

### 3. 输出必须可对账

最终答案不能只是摘要。必须让用户能看清：

- 查过哪些搜索面
- 找到了哪些路径
- 哪些被排除，为什么排除
- 哪些仍未验证
- 如果要改，具体先改哪一条、怎么验

详细输出格式见 [references/analysis-playbook.md](references/analysis-playbook.md)；实施任务还要遵守 [references/implementation-playbook.md](references/implementation-playbook.md) 的执行与汇报要求。

## 结果优先的停止条件

只有在以下条件成立时，才可以给出“当前范围内已基本找全”或“当前批次已修复”的判断：

- 已按对应模式读完所需 reference
- 搜索面对账完整，没有把“未查”伪装成“未发现”
- 已确认路径、已排除路径和未验证边界都已编号
- 如果进入实施模式，每个已完成修复点都有验证证据

任何一步不满足，就继续分析，或明确告诉用户还缺什么，不要硬收尾。
